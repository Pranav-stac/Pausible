**Integrated Plan Page**

Report Page 10: Your Personalised Plan

Version 1.0 — June 2026

*This document provides the developer implementation specification for generating the Integrated Plan page in the Pausibl Wellness Intelligence Report. It supplements the Integrated Plan Generator Framework v1.1 with rendering logic, API structure, and AI synthesis instructions specific to the report page.*

# **1\. Page Overview**

The Integrated Plan page (Page 10\) presents the user's personalised phased wellness plan. It transforms the plan\_output object from the plan generator into a user-facing page that shows what to do, when, and why — without exposing engine internals.

**Page title: "Your Personalised Plan"**

## **1.1 Page Position in Report**

| Page | Title | Purpose |
| :---- | :---- | :---- |
| 1 | Cover | Identity \+ persona title |
| 2 | Understanding Your Personality | OCEAN radar chart \+ trait descriptions |
| 3 | Your Pattern Match | Fit score \+ blend analysis |
| 4 | Your Primary Pattern | Persona deep-dive |
| 5 | Your Secondary Pattern | Secondary persona influence |
| 6 | What You Don't See | Blind spots \+ pattern predictions |
| 7 | Key Actions | Top recommendations by pillar |
| 8 | High-Impact Priorities | Prioritised action list |
| 9 | What Comes Next | Coaching guidance |
| **10** | **Your Personalised Plan** | **Phased integrated plan (NEW)** |

*Page 10 is a NEW addition to the report. It depends on the plan\_output object generated by the plan generator engine (see Integrated Plan Generator Framework v1.1). This page is generated AFTER Pages 7–8 because it uses the same scored/filtered recommendation set.*

# **2\. Required Data Sources**

The plan page consumes data from multiple engine outputs. No new data collection is required — all inputs are already computed by the recommendation engine and plan generator.

| Data Source | Origin | Used For |
| :---- | :---- | :---- |
| plan\_output | Plan generator engine | Phase structure, anchor habits, daily/weekly items, readiness signals |
| persona | OCEAN assessment → clustering | Phase intent descriptions, progression style |
| fit\_tier | Fit score calculation | Tier-specific adjustments displayed to user |
| secondary\_persona | Blend analysis | Secondary influence notes |
| scored\_recommendations\[\] | Recommendation engine | Individual recommendation display text and persona context |
| barriers\[\] | Contextual questionnaire | Barrier override explanations |
| goals\[\] | Contextual questionnaire | Goal-based framing in AI synthesis |
| ocean\_profile | OCEAN assessment | Trait-specific plan commentary in AI synthesis |

# **3\. Page Layout Specification**

The plan page is divided into 4 sections, rendered sequentially.

## **3.1 Section A: Plan Overview Header**

A brief introduction that frames the plan. This section is AI-synthesised.

| Element | Source | Example |
| :---- | :---- | :---- |
| Plan title | Static text | "Your Personalised Plan" |
| Plan subtitle | AI synthesis | "A 10-week plan designed for your pattern — steady, cautious, built for consistency." |
| Total phases | plan\_output.total\_phases | "3 phases" |
| Total duration | Derived from phase durations | "Approximately 10 weeks" |
| Progression style | Phase Configuration Matrix | "Steady-cautious" |
| Primary goal framing | AI synthesis using goals\[\] | "Built around your goal of muscle gain and strength." |

*The plan subtitle and primary goal framing are the ONLY AI-synthesised elements in this section. Everything else is deterministic.*

## **3.2 Section B: Phase Cards**

Each phase is rendered as a card. This is the core visual element of the page. The number of cards equals plan\_output.total\_phases (2–4).

**Per-phase card structure:**

| Element | Source | Display Rules |
| :---- | :---- | :---- |
| Phase number \+ name | plan\_output.phases\[\].phase\_number \+ name | "Phase 1: Build the Floor" |
| Duration estimate | plan\_output.phases\[\].approx\_duration\_weeks | "Approximately 3 weeks" |
| Phase intent | AI synthesis using phases\[\].intent | 1–2 sentence user-facing description. Must NOT use engine terminology. |
| Anchor habit | plan\_output.phases\[\].anchor\_habit | Highlighted/emphasised. Use persona\_context resolved text. |
| Daily rhythm | plan\_output.phases\[\].daily\_rhythm\[\] | Listed as daily actions. Use recommendation display\_text. |
| Weekly rhythm | plan\_output.phases\[\].weekly\_rhythm\[\] | Listed as weekly actions. Use recommendation display\_text. |
| Readiness signal | AI synthesis using phases\[\].readiness\_signal | User-facing version. Example: "You're ready for Phase 2 when your 3 weekly sessions feel automatic — not something you decide each morning." |
| Pillar indicators | plan\_output.phases\[\].pillar\_distribution | Small icons or labels showing which pillars are covered (Sleep, Nutrition, Physical, Mental). |

## **3.3 Section C: Plan Notes**

A short section explaining any adjustments the engine made. This helps users understand why their plan looks the way it does.

| Condition | Note Template | Example |
| :---- | :---- | :---- |
| Fit tier is Core/Leaning/Exploring | "Your plan has been adjusted for your \[fit\_tier\] fit — \[explanation\]." | "Your plan has been adjusted for your Leaning fit — Phase 1 has been extended to give you more time to settle in." |
| Secondary persona present | "Your secondary pattern (\[secondary\]) influences your plan — \[modification\]." | "Your secondary pattern (Shielded Turtle) influences your plan — Phase 1 focuses more on emotional comfort before action." |
| Barrier override active | "Because \[barrier\] is a priority for you, \[adjustment\]." | "Because time is limited for you, all Phase 1 actions take under 15 minutes." |
| Phase 1 density override active (N ≥ 5.5) | "Phase 1 prioritises mental comfort and emotional safety before \[goal\]." | "Phase 1 prioritises mental comfort and emotional safety before muscle gain. Your goal takes centre stage from Phase 2." |

*Show a maximum of 3 notes. If more than 3 conditions are active, prioritise: fit tier note \> barrier note \> secondary persona note \> density override note. The notes are AI-synthesised using the generation\_notes field from plan\_output.*

## **3.4 Section D: What This Plan Does Not Do**

A brief closing disclaimer, always present. This is static text, not AI-synthesised.

"This plan is not a medical prescription or a fixed programme. It adapts to how you respond. The durations are estimates — your readiness signals matter more than the calendar. If something isn't working, that's useful information, not failure."

# **4\. API Call Specification**

The plan page requires 1 API call to the AI synthesis endpoint. This is a separate call from the existing 8 report API calls.

## **4.1 API Call: Plan Page Synthesis**

**Call ID: A14**

**Model: GPT 5.4 mini**

**Max output tokens: 800**

**Temperature: 0.4**

### **4.2 System Prompt**

You are a wellness plan writer for Pausibl. You write warm, clear, personalised plan descriptions that make the user feel understood and motivated.

Rules:  
\- Never use OCEAN trait names (Openness, Conscientiousness, Extraversion, Agreeableness, Neuroticism). Use user-facing names: Openness, Discipline, Social Energy, Agreeableness, Stress Sensitivity.  
\- Never use persona animal names (Turtle, Deer, Fox, Wolf, Bear, Elephant). Describe the pattern without naming the animal.  
\- Never use engine terminology (activation energy, fit score, blend ratio, pillar distribution, readiness signal).  
\- Never use motivational clichés ("crush your goals", "unleash your potential", "transform your life").  
\- Write in second person ("you", "your").  
\- Keep each synthesised section to 1–2 sentences maximum.  
\- If the user's stated goals imply intensity or timelines misaligned with their behavioural pattern, acknowledge the goal positively and frame the plan as building toward it in stages. Never reject or label a goal as unrealistic.

### **4.3 User Prompt Template**

Generate plan page content for the following user:

Persona: {persona\_display\_name}  
Fit tier: {fit\_tier}  
Secondary persona: {secondary\_persona\_display\_name} ({blend\_percentage}%)  
Goals: {goals\_list}  
Barriers: {barriers\_list}  
Total phases: {total\_phases}  
Total duration: {total\_duration\_weeks} weeks

Phase details:  
{for each phase:}  
Phase {phase\_number}: {phase\_name}  
Duration: {approx\_duration\_weeks}  
Intent (engine): {intent}  
Anchor habit: {anchor\_habit\_display\_text}  
Readiness signal (engine): {readiness\_signal\_description}  
{end for}

Generation notes: {generation\_notes}

Generate:  
1\. plan\_subtitle: One sentence describing the overall plan tone and structure.  
2\. goal\_framing: One sentence connecting the plan to the user's primary goal.  
3\. For each phase:  
   a. phase\_intent\_user: 1–2 sentence user-facing description (rewrite the engine intent in warm, clear language).  
   b. readiness\_signal\_user: 1 sentence translating the readiness signal into what the user will feel/experience.  
4\. plan\_notes: Array of 1–3 short sentences explaining key adjustments (fit tier, secondary persona, barriers).

Return as JSON.

### **4.4 Expected Response Schema**

{  
  "plan\_subtitle": "string",  
  "goal\_framing": "string",  
  "phases": \[  
    {  
      "phase\_number": 1,  
      "phase\_intent\_user": "string",  
      "readiness\_signal\_user": "string"  
    }  
  \],  
  "plan\_notes": \["string"\]  
}

# **5\. Blocklist Enforcement**

The plan page must enforce the same blocklist as all other report pages. Additionally, the plan page has its own restricted terms that must never appear in user-facing content.

## **5.1 Global Blocklist (applies to all pages)**

| Category | Blocked Terms | Replacement |
| :---- | :---- | :---- |
| OCEAN trait names | Conscientiousness, Neuroticism, Extraversion | Discipline, Stress Sensitivity, Social Energy |
| Persona animals | Turtle, Deer, Fox, Wolf, Bear, Elephant | Describe the pattern without naming the animal |
| Engine jargon | fit score, blend ratio, centroid, OCEAN, persona | Do not use; rephrase |

## **5.2 Plan-Specific Blocklist**

| Blocked Term | Reason | Alternative |
| :---- | :---- | :---- |
| activation energy | Engine-internal scoring concept | Do not reference; the plan structure already reflects this |
| readiness signal | Engine terminology | "You'll know you're ready when..." |
| pillar distribution | Engine-internal weighting | "Your plan covers sleep, nutrition, movement, and mental wellness" |
| phase assignment | Engine process | Do not reference |
| density weighting | Engine scoring | Do not reference |
| core / supporting / optional / conditional | Recommendation strengths | Do not label recommendations by strength to the user |
| recommendation master | Internal data source | Do not reference |
| scoring formula | Engine calculation | Do not reference |

*Blocklist must be enforced BOTH in the AI synthesis prompt (via system prompt instructions) AND as a post-generation validation step. The app logic must scan all AI-generated text for blocked terms before rendering.*

# **6\. Deterministic vs AI-Synthesised Content**

Clear separation between what the app renders directly from data and what requires AI synthesis.

| Element | Source Type | Notes |
| :---- | :---- | :---- |
| Page title | Deterministic | Always "Your Personalised Plan" |
| Plan subtitle | AI synthesis | 1 sentence |
| Goal framing | AI synthesis | 1 sentence |
| Phase number \+ name | Deterministic | From plan\_output |
| Phase duration | Deterministic | From plan\_output |
| Phase intent (user-facing) | AI synthesis | Rewrite of engine intent |
| Anchor habit text | Deterministic | From recommendation display\_text with persona\_context resolution |
| Daily rhythm items | Deterministic | From recommendation display\_text |
| Weekly rhythm items | Deterministic | From recommendation display\_text |
| Readiness signal (user-facing) | AI synthesis | Translation of engine signal |
| Pillar indicators | Deterministic | From pillar\_distribution |
| Plan notes | AI synthesis | 1–3 sentences from generation\_notes |
| Disclaimer | Deterministic | Static text (Section 3.4) |

*Total AI-synthesised elements: plan subtitle (1) \+ goal framing (1) \+ phase intents (2–4) \+ readiness signals (2–4) \+ plan notes (1–3) \= approximately 7–12 short text blocks. All are 1–2 sentences. The page is majority deterministic.*

# **7\. Persona Context Resolution for Plan Items**

Each recommendation in the plan has persona-specific display text. The same resolution logic from the main report applies (see Developer Implementation Guide v2.1, Section 11).

**Resolution priority:**

* 1\. Check the primary persona's context column for the recommendation. If non-empty, use it.  
* 2\. If empty and secondary persona exists (blend \> 15%), check the secondary persona's context column.  
* 3\. If both are empty, use the recommendation's default text field.

*This is identical to the existing persona context resolution in Pages 7–8. No new logic is required — reuse the same resolution function.*

# **8\. Rendering Rules**

## **8.1 Phase Card Styling**

| Rule | Specification |
| :---- | :---- |
| Phase card layout | Each phase is a distinct visual card with clear separation |
| Anchor habit emphasis | Bold or highlighted — visually distinct from other items |
| Daily vs weekly separation | Two sub-sections within each card: "Daily" and "Weekly" |
| Phase progression | Visual indicator showing progression (Phase 1 → 2 → 3 → 4\) |
| Inactive phases | All phases are shown but only Phase 1 is marked as "Start here". Later phases show duration estimates but are visually muted. |
| Pillar coverage | Small colour-coded icons per pillar (use existing pillar colours from Pages 7–8) |
| Readiness signal | Displayed at the bottom of each card in a distinct visual style (e.g., lighter background, italicised) |

## **8.2 Content Length Limits**

| Element | Max Length | Enforcement |
| :---- | :---- | :---- |
| Plan subtitle | 120 characters | AI prompt \+ post-validation truncation |
| Goal framing | 100 characters | AI prompt \+ post-validation truncation |
| Phase intent (user-facing) | 200 characters | AI prompt \+ post-validation truncation |
| Readiness signal (user-facing) | 150 characters | AI prompt \+ post-validation truncation |
| Individual plan note | 120 characters | AI prompt \+ post-validation truncation |
| Recommendation display text | As per recommendation master | Already enforced in existing pipeline |
| Anchor habit text | As per recommendation master | Already enforced in existing pipeline |

# **9\. Error Handling**

| Scenario | Behaviour |
| :---- | :---- |
| AI synthesis call fails | Render the page with deterministic content only. Use engine intent text (not user-facing) as fallback for phase descriptions. Log the failure. |
| AI response contains blocked terms | Strip the blocked term and replace with generic alternative. Log the violation for monitoring. |
| AI response exceeds character limits | Truncate at the nearest sentence boundary. Never truncate mid-sentence. |
| Plan has 0 phases | This should never happen. If it does, skip the plan page entirely and log a critical error. |
| A phase has 0 daily items | Render the phase card without a "Daily" sub-section. This is valid for Shielded Turtle Phase 1\. |
| Anchor habit is missing | Use the highest-scoring item in the phase as a visual anchor. Log the gap. |
| AI returns invalid JSON | Retry once with the same prompt. If retry fails, fall back to deterministic-only rendering. |

# **10\. Integration with Existing Pipeline**

The plan page slots into the existing report generation pipeline as an additional step after the recommendation engine output is complete.

## **10.1 Processing Order**

| Step | Process | Output | Status |
| :---- | :---- | :---- | :---- |
| 1 | OCEAN assessment scoring | ocean\_profile | Existing |
| 2 | Persona assignment \+ fit score | persona, fit\_tier, secondary\_persona | Existing |
| 3 | Contextual questionnaire processing | barriers\[\], goals\[\], context\_tags | Existing |
| 4 | Recommendation scoring \+ filtering | scored\_recommendations\[\] | Existing |
| 5 | Recommendation ranking \+ selection | selected\_recommendations (Pages 7–8) | Existing |
| 6 | Clustering for AI synthesis | clustered\_recommendations | Existing |
| 7 | AI synthesis (Pages 1–9) | 8 API calls | Existing |
| **8** | **Plan generator engine** | **plan\_output** | **NEW** |
| **9** | **Plan page AI synthesis** | **1 API call (A14)** | **NEW** |
| **10** | **Plan page rendering** | **Report Page 10** | **NEW** |

*Steps 8–10 are additive. They do not modify any existing pipeline steps. The plan generator engine (Step 8\) operates on the same scored\_recommendations\[\] output from Step 4 — it does not re-score or re-filter.*

## **10.2 Dependencies**

| New Component | Depends On | Does NOT Depend On |
| :---- | :---- | :---- |
| Plan generator engine | persona, fit\_tier, secondary\_persona, scored\_recommendations\[\], barriers\[\], goals\[\] | AI synthesis output from Pages 1–9 |
| Plan page AI synthesis (A14) | plan\_output from plan generator | Any other AI synthesis call (can run in parallel with A1–A8) |
| Plan page rendering | plan\_output \+ A14 response | Rendering of Pages 1–9 |

*The plan page is independent of Pages 1–9 rendering. It can be generated in parallel with the main report once the recommendation engine output is available. The only serial dependency is: recommendation engine → plan generator → plan page AI synthesis → plan page render.*

# **12\. Risks and Safeguards**

| Risk | Impact | Safeguard |
| :---- | :---- | :---- |
| AI synthesis uses blocked terms | User sees engine jargon ("activation energy", persona animals) | Post-generation blocklist scan \+ replacement. Log violations for prompt tuning. |
| Phase 1 dominated by goal-misaligned content | High-N user gets Physical 35% in Phase 1 instead of Mental safety | Phase 1 density override for N ≥ 5.5. Deterministic enforcement, not AI-dependent. |
| Plan feels generic | User doesn't see personalisation | Persona context resolution ensures recommendation text is persona-specific. AI synthesis uses persona \+ barrier \+ goal context. |
| Too many plan notes | Information overload | Cap at 3 notes. Prioritise by: fit tier \> barrier \> secondary persona \> density override. |
| Phase count mismatch | Rendering shows wrong number of phases | plan\_output.total\_phases is authoritative. Render exactly that many cards. |
| Recommendation text too long for card layout | Card overflow / truncation | Use existing recommendation display\_text limits. If still too long, truncate at last complete word. |
